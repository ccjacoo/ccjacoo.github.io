    学习：

    程序访问通常是通过高层次的API接口而不是直接进行系统调用。

    三种常用的应用程序编程接口：Win32 API用于 Windows， POSIX API用于 Linux和Unix系统，JAVA API用于JAVA虚拟机（JVM）。

    系统调用（int&iret）的开销大于函数调用（call&ret）。需要引导机制，创建内核堆栈，然后验证参数，内核态会映射到用户态的地址空间，TLB的内容会实效。

    ​ ucore系统调用read：用户态遇到一个 int 软中断，然后进入到内核里来，运行一段汇编程序叫做 alltraps()，在这个程序中会获得中断所需要的信息组成的数据结构，即 TF（trapframe）数据结构。 trap()函数其中有系统调用对应的中断向量： tf->trapno== T_SYSCALL，然后转到系统调用的这个syscall()函数里，然后读取eax：tf->tf_regs.reg_eax==SYS_read，读取到的系统调用编号是read的，所以知道要执行read操作。于是随后继续从 tf->sp中获取 fd，buf，length等需要用到的参数。最后到sysfile_read()函数里去操作底层的驱动程序，实现读取文件的功能。返回的时候到trapret()中来，最后部分的ireturn会把读到的返回值长度返回给用户态。

    1.任务1：

    1.1练习1：

    理解通过 make 生成执行文件的过程。约定写在Makefile中。

    命令 ：

    make v=

    就可以详细展开编译的过程。

    无非是一层层编译，然后最后通过dd命令组合生成的elf文件，生成镜像文件。主要是依靠Makefile文件里所规定的规则。比如它是通过tools/sign.c文件来修饰编译生成的bootloader文件。再比如当make debug时，它通过tools/gdbinit文件来载入gdb的命令。

    Makefile通过一系列命令生成了bootblock和kernel这两个elf文件，之后通过dd命令将bootblock放到第一个sector，将kernel放到第二个sector开始的区域。可以明显看出bootblock就是引导区，kernel则是操作系统内核。

    ucore.img的生成过程为：

    1.编译所有生成 bin/kernel所需的c文件（在 kern文件夹下）得到对应的o文件

    2.ld链接所有o文件，生成 bin/kernel

    3.继续编译bootasm.s，bootmain.c和sign.c

    4.根据sign规范生成 obj/bootlock.o

    5.最后生成ucore.img

    Makefile文件相应内容如下：

    #————–create ucore.img—————

    UCOREIMG := $(call totarget,ucore.img)

    $(UCOREIMG): $(kernel) $(bootblock)
    $(V)dd if=/dev/zero of=$@ count=10000
    $(V)dd if=$(bootblock) of=$@ conv=notrunc
    $(V)dd if=$(kernel) of=$@ seek=1 conv=notrunc

    $(call create_target,ucore.img)

    主引导扇区的特征：大小为512字节，倒数两个字节为 0x55AA，其余空间填0.

    总结：由不超过466字节的启动代码和不超过64字节的硬盘分区表加上两个字节的结束符组成。

    1.2 练习2：

    题目1：从CPU加电后执行的第一条指令开始，单步跟踪BIOS的执行

    BIOS是运行在实模式下的，该模式下地址为16位偏移量加上4位段基址，所以gdb调试的时候应该注意：由于ucore为i386的操作系统，gdb调试时默认是运行在i386模式下，需要用set arch i8086，才能正确反汇编十六位的代码

    修改lab1的tools下的gdbinit文件，修改为：

    set architecture i8086

    target remote :1234

    然后编译 make debug，得到：

    warning: A handler for the OS ABI "GNU/Linux" is not built into this configuration
of GDB.  Attempting to continue with the default i8086 settings.

The target architecture is assumed to be i8086
0x0000fff0 in ?? ()
(gdb) x/2i $pc
=> 0xfff0:      add    %al,(%bx,%si)
   0xfff2:      add    %al,(%bx,%si)
//刚加电，进去第一步就是来到 0xfff0 处，这是由于BIOS的第一条指令总是位于0xf000:0xfff0这个地址
//然后我们根据题目要求，si单步执行
(gdb) si
0x0000e05b in ?? ()
(gdb) x/2i $pc
=> 0xe05b:      add    %al,(%bx,%si)
   0xe05d:      add    %al,(%bx,%si)
//可以看到下一步执行来到了0xe05b。 0xf000:0xfff0这个地址是一条跳转指令，所以跳转到0xf000:0xe05b这个地址。 从这里开始就是bios的代码了。

    题目3：从0x7c00开始跟踪代码运行,将单步跟踪反汇编得到的代码与bootasm.S和 bootblock.asm进行比较
