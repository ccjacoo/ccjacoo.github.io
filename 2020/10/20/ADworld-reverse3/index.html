    2. serial-150

    64位ELF文件，ida静态调试看不出什么信息，开启动态调试，我用的是gdb，当然也可以用ida远程连接linux进行动态调试。

    一步步运行下去，根据多次调试，我们把断点下在 0x400a0b 处，因为在这里需要我们输入东西。

    b *0x400a0b

    然后一步步运行下去，

    可以看到当前，首先call strlen函数，返回的rax存储长度，然后和 16 (0x10) 比较，长度相同跳转到 0x400a3c

    由于我刚好输入长度为16的字符串，所以认证通过。

    0x400a3c语句把 [rbp-0x200]的内容给 eax，然后让al和 0x45 比较，

    可惜我这里是 ’a‘ (0x61)，看来应该是 0x45 ，即 ‘E’， 好吧，前面流程重新来过，输入 16个 ‘E’，然后跳转到 0x400a55

    可以看到它从内存中取出东西分别放在edx和eax中，然后

    然后eax = eax + edx ，与 0x9b 比较，这里edx是之前的0x45，所以这里的eax = 0x9b - 0x45 = 0x56 ，即 ‘ V ‘

    [rbp-1f1]是字符串尾部，所以现在我们知道了flag 的首尾字符分别是 E 和 V。

    然后就这样继续慢慢调试下去，这题的意思是 从头开始取一字符进行比较，比较通过检验后，从后往前取一字符，然后判断这两个数的和是否为正确，两两一组逐渐往里缩。

    肝疼…….我直接看答案了。

    flag: EZ9dmq4c8g9G7bAV

    后续看了其他人的评论，才知道里面插入了很多的0eb的花指令，干扰了IDA的分析，nop掉就行。

    5. gametime

    32位，运行程序，是个游戏，发现在一定时间内不输入和输错都会引起程序闪退,并打印关键语句： “ UDDER FAILURE! http://imgur.com/4Ajx21P “。而且到后面所给的时间越来越短，来不及输入就闪退。

    ida打开搜索字符串，定位该关键语句，然后跟进，有两处调用，分别是sub_401507和sub_401435

    分析：首先v7是经过随机数生成然后mod22得到的，所以取值范围在 0~21，

    v10的值为10，那么while循环的次数也就是10，

    注意到这里：

    if ( v6 != (v9 ^ v5) )
      v11 = 0;

    要使得这个条件成立才可，不然v11为0就game over 了。

    v6数组是已知的字符串，v9的值从0开始，加12345，然后在循环之中不断乘1828812941然后加12345………

    v5是我们的输入，不过只循环10次，也就是取10个字符。

    破案了，这道题实际就是取10位我们输入的字符，异或后，与byte_6B4270数组比较是否相同。

    写出脚本:

    index = [0x5F,0xF2,0x5E,0x8B,0x4E,0x0E,0xA3,0xAA,0xC7,0x93,0x81,0x3D,0x5F,0x74,0xA3,0x09,
0x91,0x2B,0x49,0x28,0x93,0x67]
flag = ''
for i in range(22):   #爆破
    v6 = i + 1
    v10 = 0
    v11 = 0   #初始值为0
    while v10 < v6:
        v10 = v10 + 1
        v11 = 1828812941 * v11 + 12345
    flag += chr((index[i]^v11)&0xff)
print (flag)

    得到flag：flag{d826e6926098ef46}

    至于脚本中为什么还要 &0xff 呢，那是因为源程序中 (show casts展开) 实际显示是

    if ( v6 != ((unsigned __int8)v9 ^ v5) )

    v9的类型是unsigned_int8，只保留8位 （我们这个是64位程序，有16位），所以需要 &0xff

    哇太细节了吧，根本没注意。
