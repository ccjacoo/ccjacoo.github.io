    延迟绑定：

    我们知道在Linux中如果程序想要调用其他动态链接库的函数，必须要在程序加载的时候动态链接；

    在一个程序运行过程中，可能很多函数在程序执行完时都不会用到，比如一些错误处理函数或者一些用户很少用到的功能模块；并且如果可执行文件调用的动态库函数很多时，那在进程初始化时都对这些函数做地址解析和重定位工作的话，会大大增加进程的启动时间。
    所以ELF采用一种叫做延迟绑定(Lazy Binding)的做法，只有动态库函数在被调用时，才会地址解析和重定位工作。基本思想就是当函数第一次被调用的时候才进行绑定(符号查找、重定位等)；

    运用到延迟绑定的场景有：

    1. fork之后父子进程内存的写时拷贝机制
    2. Linux用户态内存空间分配与物理内存分配机制
    3. C++库的string类写时拷贝机制
    4. 动态链中的延迟重定位机制

    而在Linux 中是利用 _dl_runtime_resolve(link_map_obj, reloc_index)函数来对动态链接的函数进行重定位的；

    _dl_runtime_resolve函数运行过程：

    首先总的来说一下_dl_runtime_resolve函数如何使程序第一次调用一个函数：

    1.首先用link_map访问.dynamic，分别取出.dynstr、 .dynsym、 .rel.plt的地址；
    2..rel.plt + 参数reloc_index,求出当前函数的重定位表项Elf32_Rel的指针，记作rel;
    3.rel->r_info >> 8作为.dynsym的下标，求出当前函数的符号表项Elf32_Sym的指针，记作sym;
    4..dynstr + sym->st_name得出符号名字符串指针;
    5.在动态链接库查找这个函数的地址，并且把地址赋值给*rel->r_offset，即GOT表;
    6.最后调用这个函数；

    具体流程：

    以XDCTF2015的pwn200中的strlen函数为例。
    我们想要在strlen函数处下断点，则需要找到call strlen 汇编代码处：

    objdump -d bof | grep call

    得到

     804835c:    e8 bf 00 00 00           call   8048420 <__x86.get_pc_thunk.bx>
 8048371:    e8 6a 00 00 00           call   80483e0 <write@plt+0x10>
 804840c:    e8 af ff ff ff           call   80483c0 <__libc_start_main@plt>
 8048453:    ff d0                    call   *%eax
 804848d:    ff d2                    call   *%edx
 80484af:    e8 7c ff ff ff           call   8048430 <deregister_tm_clones>
 80484e0:    ff d2                    call   *%edx
 80484fe:    e8 8d fe ff ff           call   8048390 <setbuf@plt>
 8048514:    e8 87 fe ff ff           call   80483a0 <read@plt>
 8048579:    e8 12 fe ff ff           call   8048390 <setbuf@plt>
 8048588:    e8 23 fe ff ff           call   80483b0 <strlen@plt>
 804859a:    e8 31 fe ff ff           call   80483d0 <write@plt>
 80485a2:    e8 44 ff ff ff           call   80484eb <vuln>
 80485c4:    e8 57 fe ff ff           call   8048420 <__x86.get_pc_thunk.bx>
 80485dc:    e8 77 fd ff ff           call   8048358 <_init>
 8048604:    ff 94 bb 08 ff ff ff     call   *-0xf8(%ebx,%edi,4)
 8048628:    e8 f3 fd ff ff           call   8048420 <__x86.get_pc_thunk.bx>

    可以得知 call strlen@plt 地址为 0x8048588

    gdb调试，在0x8048588处下断点，然后 si 单步步入 call strlen@plt

    我们看到程序没有直接转到strlen函数，而是jmp到0x804a014 (GOT表)

    =>  0x80483b0 <strlen@plt>:    jmp    DWORD PTR ds:0x804a014   //jmp到0x804a014
    0x80483b6 <strlen@plt+6>:    push   0x10
    0x80483bb <strlen@plt+11>:    jmp    0x8048380

    我们查看一下0x804a014存的是什么内容：

    gdb-peda$ x/x 0x804a014
0x804a014:    0x080483b6

    由于0x804a014中存的是 0x080483b6(即当前指令的下一条)，所以jmp到0x804a014后又回到当前指令的下一条继续执行。即执行 push 0x10
    push 0x10入栈后，就jmp 到 0x8048380 （公共plt表项地址）

    => 0x8048380:    push   DWORD PTR ds:0x804a004
   0x8048386:    jmp    DWORD PTR ds:0x804a008

    可以看到又push了0x804a004入栈。
    之后查看一下0x804a004：

    gdb-peda$ x/x 0x804a004
0x804a004:    0xf7ffd918

    然后jmp到 0x804a008 （这里就是dl_runtime_resolve的地址）

    总之上面流程中 不停jmp，然后push了一些参数，最终跳转到了_dl_runtime_resolve函数。

    回观之前的操作，我们在自己的plt表项中push了0x10,又在公共plt表项中push了0x804a004，这两个东西就是dl_runtime_resolve的两个参数。
    现在：
    reloc_arg —-> 0x10
    link_map —-> 0xf7ffd918

    _dl_runtime_resolve(link_map, reloc_arg)

    到现在还只是第一步。

    接下来，通过link_map地址我们就可知晓 .dynamic 地址：

    gdb-peda$ x/10wx 0xf7ffd918
0xf7ffd918:    0x00000000    0xf7ffdc04    0x08049f14    0xf7ffdc08

    得到的第三个地址–0x08049f14 就是 .dynamic 地址
    然后
    再通过.dynamic地址我们可以找到 .dynstr / .dynsym / .rel.plt 地址：

    gdb-peda$ x/40wx 0x08049f14
0x8049f14:    0x00000001    0x00000001    0x0000000c    0x08048358
0x8049f24:    0x0000000d    0x08048624    0x00000019    0x08049f08
0x8049f34:    0x0000001b    0x00000004    0x0000001a    0x08049f0c
0x8049f44:    0x0000001c    0x00000004    0x6ffffef5    0x080481ac
0x8049f54:    0x00000005    0x08048278    0x00000006    0x080481d8
0x8049f64:    0x0000000a    0x0000006b    0x0000000b    0x00000010
0x8049f74:    0x00000015    0xf7ffd904    0x00000003    0x0804a000
0x8049f84:    0x00000002    0x00000028    0x00000014    0x00000011
0x8049f94:    0x00000017    0x08048330    0x00000011    0x08048318
0x8049fa4:    0x00000012    0x00000018    0x00000013    0x00000008

    .dynamic地址偏移0x44的位置（.dynamic + 0x44 =0x8049f58）存储着.dynstr的地址：
    .dynstr ==> 0x08048278

    .dynamic地址偏移0x4c的位置（.dynamic + 0x4c =0x4c）存储着.dynsym的地址：
    .dynsym ==> 0x080481d8

    .dynamic地址偏移0x84的位置（.dynamic + 0x84 =0x8049f98）存储着.rel.plt的地址：
    .rel.plt ==> 0x08048330

    因此可得：
    ELF32_Rel = .rel.plt + reloc_arg = 0x08048330 + 0x10 = 0x8048340 ,记作rel
    通过rel可得到两个信息：

    gdb-peda$ x/10x 0x8048340
0x8048340:    0x0804a014    0x00000407    0x0804a018    0x00000507

    0x8048340这一行的前两个地址，第一个地址即为r_offset ,第二个地址为 r_info
    所以：
    r_offset ===> 0x804a014 //指向GOT表的指针
    r_info ===> 0x00000407

    再将r_info 右移8：
    0x407 >> 8 ，结果得到4 ,作为 .dynsym 的下标，即.dynsym[4]

    gdb-peda$ x/30x 0x080481d8
0x80481d8:    0x00000000    0x00000000    0x00000000    0x00000000   //.dynsym[0]
0x80481e8:    0x00000033    0x00000000    0x00000000    0x00000012   //.dynsym[1]
0x80481f8:    0x00000027    0x00000000    0x00000000    0x00000012   //.dynsym[2]
0x8048208:    0x00000052    0x00000000    0x00000000    0x00000020   //.dynsym[3]
0x8048218:    0x00000020    0x00000000    0x00000000    0x00000012   //.dynsym[4]

    在下标为4即.dynsym[4]处得到函数名称的偏移： name_offset = 0x20

    然后 st_name = .dynstr + name_offset = 0x08048278 + 0x20 = 0x8048298
    st_name为符号字符串名

    查看st_name地址：

    gdb-peda$ x/s 0x8048298
0x8048298:    "strlen"

    这样就找到了strlen函数的字符串名。

    最后，在动态链接库查找这个函数的地址即可，并把地址赋给 *rel -> offset ，即GOT表

    以上便是dl_runtime_resolve 函数解析的工作原理。实际上它是通过最后的 st_name 来确定执行那一个函数的。

    关于动态链接的关键section：

    .dynamic：

    dynamic一般保存了ELF文件依赖于哪些动态库,动态符号节信息,动态符号节信息;
    .dynamic是GOT表的第一项；而GOT表的第二项是link_map的地址，第三项是_dl_runtime_resolve函数的地址，第四项开始才是函数的GOT表;

    .dynsym：

    查看.dynsym基地址和内容：

    objdump -s -j .dynsym ./filename

    动态链接的 ELF 文件具有专门的动态符号表，其使用的结构就是 Elf32_Sym，但是其存储的节为 .dynsym
    结构体中的域有：

    Elf32_Word    st_name; //是相对于.dynstr段开始的偏移值，那里有这个符号名字的字符串。
Elf32_Addr    st_value;//如果该符号被导出，则存有这个导出函数的虚拟地址，否则为NULL
unsigned char st_info; //对于导入函数符号而言，它是0x12

    .dynsym 是运行时所需的，ELF 文件中 export/import 的符号信息全在这里

    .dynstr：

    查看.dynstr基地址和内容：

    objdump -s -j .dynstr ./filename

    此节区包含用于动态链接的字符串，大多数情况下这些字符串代表了与符号表项相关的名称；

    .rel.plt:

    查看.rel.plt基地址和内容:

    objdump -s -j .rel.plt ./filename

    这里是重定位表，也是一个结构体数组，每个项对应一个导入函数；
    结构体中的域有：

      Elf32_Addr    r_offset;   //这个值就是got表的虚拟地址 ,存储的是重定位解析后的符号地址被写入内存中的位置 （绝对地址）
  Elf32_Word    r_info;     //.dynsym节区符号表索引 ,此域的高位3字节作为一个无符号的下标，表示这个符号在 .dynsym段中的位置。

    总结：

    .rel.plt 节   函数重定位
.rel.dyn 节 变量重定位
.got节 保存全局变量偏移表
.got.plt 节 保存全局函数偏移表    ，.got.plt对应着ELF32_REL结构中的r_offset的值
.dynsym 节 包含了动态链接符号表
.dynstr 节包含了动态链接的字符串   ，这个节以\x00作为开始和结尾，中间每个字符串也以\x00间隔
.plt 节 是过程链接表，把位置独立的函数调用重定向到绝对位置
