    重写了一下，如下：

        .pos 0
    irmovq stack,%rsp    #根据栈的地址设置好rsp

    halt

    .align 8
# Source block

    .quad 0x00a
    .quad 0x0b0
    .quad 0xc00

    .quad 0x111
    .quad 0x222

    irmovq src,%rdi

    xorq %rax,%rax        #rax清0
    irmovq $1,%r14        #步长=1
    irmovq $8,%r13        #len=8
    pushq %rbx            #由于下方需要用到rdx，所以先把原先的rdx的值入栈保存起来
    andq %rdx,%rdx        #按位与操作
    jmp test
loop:
    mrmovq (%rdi),%rbx    #每一个数终将暂存于rbx中
    addq %r13,%rdi        # rdi=rdi+r13
    rmmovq %rbx,(%rsi)    #copy src data to dest
    addq %r13,%rsi        # rsi=rsi+r13
    xorq %rbx,%rax        #求每一个数的xor值，结果保存在rax中
    subq %r14,%rdx        # rdx=rdx-r14 。即 len--

test:
    jne loop
    popq %rbx             #恢复init的ebp栈帧
    ret                   #返回到main

    .pos 0x200      #设置栈的位置
stack:

    2.PartB：

    实验文件在 sim/seq 目录下，目标是想要让我们对SEQ处理器进行扩展，使其支持iaddq指令

    首先，结合irmovl、opl的计算过程，通过分析，我们得到iaddl指令的计算过程如下：

    iddal:
    fetch: #取指
        icode:ifun  <-- M1[PC]
        rA:rB  <-- M1[PC+1]
        valC <-- M4[PC+2]
        valP <-- PC+6
    decode: #译码
        valB <-- R[rB]
    execute: #执行
        valE <-- valB+valC
    memory: #访问内存
    write_back: #写回
        R[rB] <-- valE
    PC update: #更新PC
        PC <-- valP

    然后我们修改 *seq-full.hcl *文件，添加iaddq指令，。

    学习：

    csapp（2）：P25开始

    存储器有（容量capacity从小到大排列）： 寄存器，SRAM，DRAM，硬盘。

    我们需要一种存储容量又大，存储速度又快，价格又便宜的存储器。

    所以构成了层次化结构的存储器（金字塔结构）。CPU内部有寄存器和cache，CPU外部是主存（RAM和ROM），主机外部有外存（硬盘，光盘）以及后备存储器（磁带，光盘库），

    相邻的存储之间是要以等长的块（block）进行交换的。所以数据就是在相邻两层之间复制传送的。

    程序访问局部化的原因：指令是按序存放，地址连续，循环程序或子程序段重复执行；数据是连续存放的，数组元素重复，按序访问。

    时间局部性：刚被访问过的单元可能不久又会被访问，频繁访问（多发生于循环中）。所以要让最近访问过的信息保留在靠近CPU的存储器当中。

    空间局部性：刚被访问过的单元的邻近单元很可能不久被访问（数组结构，循环等）。所以也要让最近邻近单元调到靠近CPU的存储器当中。

    引入cache（高速缓存）：

    cache是处于CPU和主存之间的小容量高速缓冲存储器，由SRAM组成，其中存放最活跃（被频繁访问）的程序和数据，其实是主存块中对应内容的副本。直接制作在CPU芯片内，速度与CPU一样快。

    程序运行后，要使用的一部分数据或指令会预先拷贝在cache中。

    由于程序访问局部性特征，CPU大多数情况下直接从高速缓存中取得指令和数据，称为命中（hit），而不必访问主存。提高了效率和速度。若cache中找不到所要访问的信息则称为缺失或失靶（miss），则需要访问内存了。

    例子：有两段程序段，程序段A中是对二维数组的访问是以行，即a[i][j]进行的。而程序段B则反过来，是以列为准，a[j][i]。虽然代码空间复杂度一样，但是执行速度相比，A比B快了21倍！！！！

    程序段A的访问顺序为 a[0][0]，a[0][1]…….，而在内存中数组也是按照这个顺序存放的，故访问顺序和存放顺序一致，故空间局部性好！但由于每个a[i][j]只被访问一次，所以时间局部性差。

    当访问到a[0][0]的时候，程序会把a[0][0]邻近的数组元素都会搬到cache里，所以访问速度将明显加快。而程序段B的访问是跳着来的，当程序把a[0][0]邻近的数组元素搬到cache时，却又要求去访问a[1][0]，哈哈哈哈，这谁顶得住，所以在cache里找不到，则又要去内存里找，然后继续把a[1][0]邻近的数组元素搬到cache里，把原先的a[0][0]的”残党“部分又给冲刷掉了（cache很小，容不下那么多数据），于是前面就白搬了。这效率明显很低，像是故意在调戏cache似的。

    cache的实现：

    1.主存地址AD是否在cache中，如何判断命中（hit）？主存地址分为三部分，第一部分是16位的tag，接着第二部分是12位的index索引，最后是4位的块内地址。 如何判断？首先取中间12位的index索引，去cache中找到对应的行，第二步是取地址当中的高16位的tag标记，去和对应行的tag标记相比对，若相等，并且V=1，则判断是命中（hit）！

    2.淘汰问题，该把cache中哪些数据替换下来

    3.如何分块，块与块之间怎么分？主存被分为若干大小相等的主存块（Block），cache也被分成大小相同的块，称为cache行（line）或者 槽（Slot）。

    4.主存块和cache之间如何映射？

    映射有三种方法：

    4.1 直接映射：每个主存块映射到cache的固定行，所以可以直接取模mod，假如cache有16个槽，那么就把主存块块号mod 16即可。而每16块叫做一个块群。而cache中每一个槽号前还有一个标记，标记了该数据来自主存块的哪个块群。总之，拿到一个主存地址，高7位是主存标记（块群号），接下去4位是cache索引（块号），最后低9位是块内地址。例如 0000001 0001 000001100 表示的是第1块群中的1块（即第17块，17 mod 16=1）中的第12个单元。 另外标记是有效位，用V表示，v=1表示信息有效，v=0则无效。通常使V=0来冲刷cache。“冲刷cache”指令是内核指令。

    直接映射的特点：容易实现，决定是否命中的流程很简单，所以命中时间短；无需考虑淘汰问题，总是新的换旧的；不够灵活，cache存储空间得不到充分利用，换进换出的开销很大，命中率低。

    4.2 全相连，每个主存块映射到cache的任一行；主存地址分为主存块号（标记）和块内地址两部分。例如 00000001111 000001100 表示的是第15块中的第12个单元。

    全相连的特点：命中时间长，需要比对11位。

    4.3 第三种是前两种的折中，叫组相连：每个主存块映射到cache固定组中任一行。

    5.写数据时怎么保证cache和主存之间的一致性？

    6.怎么取cache中的数据？首先确认成功命中（hit），即找到对应行，且tag标记相等，且V=1。找到后，再取主存地址的第三部分-块内地址的高两位，若是32位即4字节的数（Word），则根据00,01,10,11等情况，多路选择器（Mux）会自己选择地址去取数据。
